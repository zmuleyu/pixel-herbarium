## 一、语言本地化

### 1.1 翻译质量标准

#### 基本原则
- **禁止纯机器翻译直出**：所有面向用户的文本须经母语审校（Native Review）。
- **分级处理策略**：按内容优先级分配人力与预算。

| 内容层级 | 范围举例 | 处理方式 | 质量标准 |
|---------|---------|---------|---------|
| **S 级**（核心转化路径） | 应用商店标题/描述、首屏文案、付费弹窗 | 专业本地化公司 + 母语二审 | 错误率 ≤ 0% |
| **A 级**（高频交互文本） | 主功能按钮、导航标签、Push 通知、空状态文案 | 母语译者初译 + 审校 | 错误率 ≤ 1% |
| **B 级**（辅助说明文本） | 帮助文档、设置页说明、次级弹窗 | 母语译者初译 | 错误率 ≤ 3% |
| **C 级**（低频/后台文本） | 错误码说明、日志提示、内部标签 | 机器翻译 + 人工核查关键词 | 语义无误即可 |

**量化验收指标**：
- S/A 级文本上线前须完成 100% 母语审校，留存审校记录。
- 使用翻译管理平台（推荐 Phrase 或 Lokalise）追踪翻译进度，完成率须达 **100%** 方可提测。
- 每次版本发布新增/修改文本，审校周期预留 **≥ 3 个工作日**。

### 1.2 日语文案规范

#### 语体选择
日本 APP 文案通常使用**です・ます体**（敬体），营造亲切而不失礼貌的感觉，避免过于生硬的命令式表达。

| 场景 | ❌ 不推荐 | ✅ 推荐 |
|------|---------|--------|
| 引导注册 | 「登録する」（命令感） | 「さっそく始めましょう」（邀请感） |
| 权限申请 | 「位置情報を許可」 | 「現在地の桜情報をお届けするために、位置情報へのアクセスをお願いします」 |
| 错误提示 | 「エラーが発生した」 | 「申し訳ありません、データの読み込みに失敗しました。もう一度お試しください」 |
| 付费引导 | 「プレミアムを購入」 | 「もっと便利に楽しむ、プレミアムプランへ」 |
| 空状态 | 「データなし」 | 「まだ情報がありません。桜の開花をお楽しみに🌸」 |

#### 季节性表达规范
日本文化高度重视"季節の挨拶"（季节性问候），内容中应自然融入时令表达：

- **2 月**：「春の足音が近づいています」（春天的脚步临近了）
- **3 月初**：「各地で開花便りが届き始めました」（各地开始传来开花的消息）
- **3 月下旬**：「今週末はお花見日和です」（这个周末是赏花的好天气）
- **4 月（满开）**：「桜が満開を迎えました、お見逃しなく」（樱花迎来满开，不要错过）
- **4 月中旬**：「はかない桜の美しさを、写真に残してみませんか」（把转瞬即逝的樱花之美留在照片里吧）

**量化验收指标**：
- 建立并维护**日语文案词汇表（Glossary）**，核心词汇 ≥ 50 条（含禁用词、推荐替换词）。
- 每季更新季节性文案库，上线前 **4 周**完成审校。
- 用户差评中涉及"文字おかしい"（文字奇怪）、"日本語変"类反馈，控制在总评价的 **0.5% 以下**。

### 1.3 字符与格式规范

#### 标点与数字格式
```
✅ 正确示例：
・全角括号：（）【】
・日式省略号：……（或「…」）
・数字混用：3月28日（土）17:00開始
・货币：¥1,200/月
・百分比：85%の桜スポットに対応

❌ 错误示例：
・半角括号：()[]
・英文省略号：...
・数字格式：3/28 Saturday 5:00PM
```

#### 文本长度控制
日语字符密度高，相同信息量所需字符数约为中文的 1.2 倍、英文的 0.6 倍，UI 组件需预留扩展空间。

| UI 组件 | 中文参考长度 | 日语预留空间 | 示例 |
|---------|------------|------------|------|
| 主按钮 | 4～6 字 | +20% 宽度余量 | 「お花見スポットを探す」 |
| 标签页 | 2～4 字 | +15% | 「開花情報」「マップ」「記録」 |
| Push 通知标题 | 10～15 字 | ≤ 40 字（含空格） | 「🌸 上野公園の桜が満開になりました！」 |
| Push 通知正文 | 20～30 字 | ≤ 100 字 | 「今週末の天気は晴れ。絶好のお花見チャンスです。現在地周辺のスポットを確認しましょう。」 |

**量化验收指标**：
- UI 走查（String QA）须覆盖所有支持的屏幕尺寸（最小 iPhone SE 320pt，最大 iPad 1024pt），文字截断（truncation）发生率 ≤ **2%**。
- 全角/半角混用错误率为 **0**（自动化 lint 工具检测）。

## 二、视觉本地化

### 2.1 色彩系统

#### 品牌主色方案
以下配色基于日本赏花审美与主流 APP 视觉调研制定：

| 色彩角色 | 色值 | 用途 | 参考依据 |
|---------|------|------|---------|
| **樱花粉（主色）** | `#FFB7C5` | 主按钮、品牌标识、高亮标签 | 日本樱花标准色（桜色）|
| **深樱粉** | `#E8829A` | 按钮悬停态、强调文字 | 提升对比度，满足无障碍标准 |
| **浅樱白** | `#FFF0F3` | 卡片背景、模块底色 | 衬托内容，营造轻盈感 |
| **枝绿（辅色）** | `#7BAE7F` | 正向状态指示、进度条 | 樱花枝干、自然联想 |
| **青空蓝** | `#A8D8EA` | 天气模块、地图水域 | 日本晴天意象 |
| **深墨（文字）** | `#1A1A2E` | 正文、标题 | 高可读性，区别于纯黑 |
| **浅灰（次要文字）** | `#8D8D9A` | 说明文字、占位符 | 层次区分 |

#### 禁用色与使用限制

| 色彩 | 禁用原因 | 替代方案 |
|------|---------|---------|
| 大面积正红 `#FF0000` | 在日本传统语境中与葬礼、神社祭祀相关，大面积使用会引发不适感 | 改用樱花粉或深樱粉 |
| 纯黑背景 `#000000` | 视觉压抑，与樱花轻盈主题相悖 | 使用深墨 `#1A1A2E` |
| 荧光黄 `#FFFF00` | 与品牌气质不符 | 用于系统级警告时保留，品牌内容禁用 |

#### 无障碍对比度要求
遵循 **WCAG 2.1 AA 标准**（日本政府数字服务指南同步引用此标准）：
- 正文文字对比度 ≥ **4.5:1**
- 大字标题（18pt+）对比度 ≥ **3:1**
- 功能性图标对比度 ≥ **3:1**

**量化验收指标**：
- 使用 Stark（Figma 插件）或 Colour Contrast Analyser 进行全量扫描，不达标项 **清零后方可上线**。
- 设计稿中禁用色出现次数 = **0**，通过设计系统（Design Token）强制约束。

### 2.2 图片与插画规范

#### 摄影内容标准
主视觉与场景图须严格筛选，标准如下：

**✅ 必须满足（Must Have）**
- 拍摄地点为**日本真实赏花场景**（推荐：上野公园、新宿御苑、嵐山、吉野山等知名地点）。
- 画面中出现日本标志性元素（和服、灯笼、鸟居、木桥等）至少 1 处。
- 图片分辨率 ≥ **2000×1500px**，JPEG 品质 ≥ 85。
- 色调与品牌色系一致（粉、白、绿为主，整体明亮清新）。

**❌ 禁止使用（Must Not）**
- 非日本场景的樱花图（武汉、首尔樱花等不可混用）。
- 带有其他国家文字或标识的图片。
- 过度修图导致失真的图片（日本用户偏好真实感）。
- 构图杂乱、光线昏暗的图片。

#### 插画风格规范
- 参考风格：**日系清新扁平**（类 NHK 气象插图、Cocoro Life 风格），或**水彩手绘质感**（适合情感类内容）。
- 人物插图：采用**抽象化/无脸化**处理（避免特定人种识别引发代入感障碍），或明确使用亚洲面孔特征。
- 禁止使用：卡通夸张变形（不符合日本成熟用户审美）、过于中国风的元素（如祥云纹、龙等）。

#### 图库资源推荐

| 资源类型 | 推荐来源 | 授权说明 |
|---------|---------|---------|
| 免费摄影图 | Unsplash、Pexels（筛选日本标签） | CC0，商用无限制 |
| 付费精品图 | Aflo（日本本土图库）、Getty Images | 按图授权，需核实商用范围 |
| 插画素材 | iStock Japan、Freepik（日系风格筛选） | 订阅制，注意授权范围 |
| 自制图标 | 基于 SF Symbols（iOS）/ Material Icons 二次定制 | 开源，需遵守设计规范 |

**量化验收指标**：
- 首发版本主视觉图 ≥ **20 张**，均通过上述 Must Have 标准核查，留存审核台账。
- 所有图片完成版权登记，无侵权风险，法务确认比例 **100%**。
- 图片加载优化：首屏主视觉 LCP（Largest Contentful Paint）≤ **2.5 秒**（4G 网络条件下）。

### 2.3 图标与 UI 组件规范

#### 应用图标（App Icon）
App Icon 是在 App Store 搜索结果中的第一印象，直接影响点击率。

**设计要求：**
- 主体元素：单枝盛开的樱花，或樱花花瓣散落构图（简洁识别度优先）。
- 背景色：渐变樱花粉（`#FFB7C5` → `#FF8FAB`）或纯白+粉色主体。
- 禁止：文字叠加（小尺寸下不可辨认）、过于复杂的构图（超过 3 个视觉元素）。
- 输出规格：按 Apple / Google 最新要求，提供 1024×1024px PNG（无 alpha 通道），由工具自动适配各尺寸。

**A/B 测试计划：**
首发前设计 **3 套**备选图标方案，通过 App Store Connect 的"产品页面优化"功能进行 A/B 测试，运行周期 ≥ **21 天**，以点击率（CTR）为核心指标选定最终方案。

#### UI 组件规范

| 组件 | 规范要求 | 量化标准 |
|------|---------|---------|
| 按钮圆角 | 主要按钮圆角 12pt，次要按钮 8pt（偏圆润，符合日系 APP 风格） | 设计令牌统一管理，偏差 = 0 |
| 触摸目标区域 | 最小点击区域 44×44pt（苹果 HIG 标准，日本用户中老年群体比例高） | 全量组件达标 |
| 加载状态 | 使用品牌色（樱花粉）的旋转动画，禁止纯灰色系统默认样式 | 100% 自定义覆盖 |
| 空状态页面 | 每个空状态须含：插画 + 说明文字 + 行动按钮，不允许纯文字空状态 | 覆盖所有功能模块的空状态 |
| 弹窗设计 | Alert 弹窗取消按钮置左、确认按钮置右（符合 iOS HIG 与日本用户习惯） | 全量弹窗遵守 |

### 2.4 动效规范

日本主流 APP（LINE、Mercari、PayPay）动效风格为**克制、流畅、有反馈感**，避免过度炫技。

| 动效类型 | 时长建议 | 缓动曲线 | 场景 |
|---------|---------|---------|------|
| 页面切换 | 250～350ms | ease-in-out | 主导航切换 |
| 弹窗出现 | 200～300ms | spring（弹性） | Alert、底部弹层 |
| 按钮反馈 | 100～150ms | ease-out | 点击态缩放 0.97 |
| 列表加载 | 300～400ms | ease-out | 骨架屏 → 内容淡入 |
| 樱花飘落特效 | 循环，单片 3～5s | cubic-bezier 自然摆动 | 特殊节点（满开提醒、成就解锁）|

**量化验收指标**：
- 所有动效在 iPhone 12（A14 芯片）上运行帧率 ≥ **60 FPS**，不出现卡顿（Jank）。
- 提供"减少动态效果"（Reduce Motion）适配，开启后所有动效替换为透明度淡入淡出。

## 三、功能本地化

### 3.1 樱花开放预测模块

这是产品核心竞争力，数据准确性直接决定用户留存。

#### 数据源标准

| 数据源 | 数据内容 | 接入优先级 | 更新频率 |
|-------|---------|----------|---------|
| **日本気象協会 tenki.jp** | 全国主要地点开花/满开预测日期 | ★★★★★（必接） | 每日更新（花期内） |
| **ウェザーニュース** | 众包用户上报开花数据，覆盖长尾地点 | ★★★★（强烈推荐） | 实时 |
| **国土交通省 国土数値情報** | 公园、绿地矢量数据（用于地图图层） | ★★★（推荐） | 年度更新 |
| **OpenWeatherMap** | 气温、降水量、风速（影响花期的气象因子） | ★★★（推荐） | 每小时 |
| **用户上报（UGC）** | APP 内用户拍照打卡的开花状态 | ★★★★（中期重点建设） | 实时 |

#### 开花状态分级显示标准

参考日本气象协会标准，定义 5 级显示状态：

```
Level 1：つぼみ（花蕾期）    ── 花苞未开，图标：闭合花蕾
Level 2：咲き始め（初开）    ── 约 10% 花朵开放
Level 3：三分咲き（三成开）  ── 约 30% 花朵开放
Level 4：満開（满开）        ── 约 80% 以上开放，核心推送节点
Level 5：散り始め（落花期）  ── 花瓣开始飘落，"花吹雪"观赏最佳期
```

**量化验收指标**：
- 全国覆盖地点数量：首发版 ≥ **200 个**，含全 47 都道府县代表性地点各至少 1 处。
- 开花预测准确率（实际开花日 vs 预测开花日误差 ≤ 3 天）≥ **85%**（以上一年度气象协会数据为基准校验）。
- 数据刷新接口响应时间 ≤ **800ms**（P95）。
- 花期高峰期（3 月 20 日 ～ 4 月 10 日）系统可用率 ≥ **99.5%**（须提前扩容）。

### 3.3 社交分享模块

#### 支持平台与分享内容规范

| 平台 | 分享格式 | 预填文案示例 | 技术实现 |
|------|---------|------------|---------|
| **LINE** | 图片 + 链接卡片 | 「🌸 {地点名}の桜が{状態}！今週末のお花見にぴったりです。詳しくはアプリで→」 | LINE SDK `shareTargetPicker` |
| **X (Twitter)** | 图片 + 文字（≤140字含链接） | 「{地点名}の桜、{状態}になりました🌸 #お花見 #{都道府県}桜 #桜2025」| Twitter Kit / 系统 Share Sheet |

| **系统相册保存** | 带水印的分享图 | — | 原生 API |

#### 分享图生成规范
生成供用户分享的图片（Share Card），须满足：
- **尺寸**：1080×1080px（方图，适配 Instagram / LINE / X）。
- **内容层**：地点名（日语）、当前开花状态、用户拍摄照片或官方图、APP Logo（右下角水印）、下载引导二维码（可选）。
- **生成时间**：用户触发"分享"操作后 ≤ **1.5 秒**完成渲染。

**量化验收指标**：
- LINE 分享完成率（触发分享 → 成功发送）≥ **90%**（含用户主动取消视为正常）。
- 分享图生成成功率 ≥ **99%**，失败时提供备用纯文字分享降级方案。

### 3.4 用户评价与留评模块

日本用户不主动留评的比例高，需主动引导但不可骚扰。

#### 留评触发时机（黄金时机原则）

```
触发条件（满足任意一条）：
✅ 用户连续使用 APP ≥ 3 天
✅ 用户累计查看花况信息 ≥ 5 次
✅ 用户完成第一次地点打卡
✅ 用户收到的开花提醒准确命中（实际开花日与预测吻合）

禁止触发的时机：
❌ APP 首次启动时
❌ 用户正在查看地图/导航过程中
❌ 用户刚刚遭遇加载失败或错误
❌ 距离上次弹出留评请求 < 30 天
```

#### 留评文案示例

```
标题：今シーズンはいかがでしたか？
正文：お花見のお役に立てていたら、ぜひ評価をお願いします。
      いただいたご意見は、より良いアプリ作りに役立てます🌸
按钮：評価する ／ あとで
```

**量化验收指标**：
- 留评提示触发后，用户完成评分的转化率目标 ≥ **15%**（行业基准约 8～12%）。
- App Store 评分目标 ≥ **4.3 星**（首发后 3 个月内）。
- 对全部用户评论进行日语回复，回复率 = **100%**，平均响应时间 ≤ **48 小时**。

### 3.5 隐私与权限申请模块

日本用户对隐私保护极为敏感，权限申请的措辞和时机直接影响通过率。

#### 权限申请话术规范

每项权限申请须遵循"**先说用途，再弹系统弹窗**"原则，在系统弹窗前展示 APP 自定义的权限说明页（Pre-Permission Screen）。

**① 位置权限（最关键）**

```
图示：地图上的樱花 PIN 图标
标题：現在地の桜情報をお届けします
说明：現在地を使って、近くの桜スポットの開花状況や
      おすすめのお花見コースをご案内します。
      位置情報は本アプリの機能提供のみに使用され、
      第三者に共有されることはありません。
按钮：[位置情報を許可する]  [今は使わない]
```

**② 通知权限**

```
图示：樱花开花的推送通知样式截图
标题：桜の満開をお見逃しなく🌸
说明：お気に入りのスポットが満開になったとき、
      天気が良いお花見日和のとき、
      タイムリーにお知らせします。
按钮：[通知を受け取る]  [あとで設定する]
```

**③ 相机与相册权限**

```
触发时机：用户点击"写真を投稿する"时
说明：桜の写真を投稿するには、カメラと写真ライブラリへの
     アクセスが必要です。
```

#### 隐私政策要求
- 隐私政策须提供**完整日语版本**，语言清晰，避免法律术语堆砌。
- 须明确说明：收集的数据类型、使用目的、第三方共享情况、用户数据删除权利。
- 遵循**个人情報保護法（2022 年改正版）**，设置数据泄露通知机制。
- 隐私政策链接须在：App Store 页面、APP 设置页、注册流程 3 处均可访问。

**量化验收指标**：
- 位置权限授权率目标 ≥ **65%**（通过 Pre-Permission Screen 提升，行业均值约 50%）。
- 通知权限授权率目标 ≥ **50%**。
- 隐私政策日语版本由具备法务背景的母语人士审校，上线前获得 **法务书面确认**。

### 3.6 客服与用户反馈模块

#### 客服体系设计

```
第一层：FAQ 自助解答（日语，≥ 30 条常见问题）
         ↓ 未解决
第二层：AI 客服（基于 LLM 的日语对话，7×24h）
         ↓ 复杂问题/负面情绪
第三层：人工客服（母语客服，工作日 10:00-18:00 JST）
```

#### 常见问题分类（FAQ 必备项）

**量化验收指标**：

- 人工客服首次响应时间（工作时间内）≤ **4 小时**。
- 用户满意度（客服体验评分）≥ **4.0/5.0**。

## 五、迭代优化机制

### 5.1 常态化复审节奏

| 节点 | 时间 | 工作内容 |
|------|------|---------|
| **年度大版本** | 每年 1 月 | 全量本地化复审、词汇表更新、花期数据源核查 |
| **花期前冲刺** | 每年 2 月下旬 | 季节性文案更新、Push 通知策略更新、压力测试 |
| **花期内热更** | 3 月～4 月 | 数据实时更新、紧急 Bug 修复（≤ 24 小时响应）|
| **花期后复盘** | 每年 5 月 | 用户反馈整理、预测准确率统计、KPI 达成分析 |
| **常规迭代** | 每 4～6 周 | 小版本功能更新，本地化同步 |

### 5.2 用户反馈驱动优化

建立从用户声音到产品迭代的闭环机制：

```
用户反馈来源：
App Store 评论 + 应用内反馈 + 客服工单 + 社媒监控（X #アプリ名 关键词）
        ↓
每周汇总分类（语言问题 / 视觉问题 / 功能问题 / 数据问题）
        ↓
按严重程度分级（P0 紧急 / P1 高优 / P2 正常 / P3 低优）
        ↓
P0：24h内修复上线热更
P1：纳入下一个 Sprint
P2/P3：排期正常迭代
        ↓
修复后在 App Store 回复中告知用户（「ご指摘いただいた件を修正しました。ありがとうございます🌸」）
```
