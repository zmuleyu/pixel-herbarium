# DeskBuddy · 输入法场景桌宠开发规划

> **项目定位**：将陆知行从「桌面独立窗口」扩展到「输入法伴随形态」，实现角色在用户工作流中的碎片化存在。
>
> **核心原则**：两种形态，一个灵魂。输入法迷你形态与桌面全身形态共享同一个情感内核。
>
> **竞品参照**：搜狗输入法 AI 宠物（2024.07 上线），借鉴其「寄生高频工具」的渠道策略，避开其「无状态聊天机器人」的产品陷阱。

## 二、分阶段开发计划

### Phase 0 · 基础设施（2 周）

**目标**：将现有情感内核从桌面形态中解耦，变成独立的数据服务层。

**具体任务**：

**0-1 情感内核独立化。** 当前情感参数（好感度、L1-L5 等级、blush/shy/gentle 情绪值、combo 计数器、衰减曲线）嵌在桌面形态的组件状态里。需要抽成独立的 EmotionCore 模块，对外暴露读写接口，支持多个渲染出口同时订阅状态变化。数据结构定义如下：

```typescript
interface EmotionState {
  affection: number;           // 0-500 好感度总值
  level: 1 | 2 | 3 | 4 | 5;  // 当前等级
  mood: {
    eye: string;               // normal | half | closed | wide | sparkle
    gaze: { x: number; y: number }; // -1~1 视线偏移
    mouth: string;             // neutral | smile | slight | open | pout
    blushCheek: number;        // 0~1 脸红程度
    blushEar: number;          // 0~1 耳尖红
    headTilt: number;          // -15~15 度
  };
  combo: {
    count: number;
    lastInteraction: number;   // timestamp
    cooldownMs: number;
  };
  decayCurves: {
    blush: { startValue: number; startTime: number; halfLife: number };
    gaze: { targetX: number; returnSpeed: number };
  };
}
```

**0-2 事件总线扩展。** 在现有事件通知系统（日历/天气/时间/闹钟）上新增 `TypingBehavior` 事件源接口。不在这个阶段实现采集逻辑，只定义事件类型：

```typescript
type TypingEvent =
  | { type: 'typing_start' }
  | { type: 'typing_stop'; idleDurationMs: number }
  | { type: 'typing_burst'; durationMs: number; speed: 'slow' | 'normal' | 'fast' }
  | { type: 'late_night_typing'; hour: number }
  | { type: 'long_session'; minutesContinuous: number }
```

**0-3 双出口分发器。** 事件触发情感状态更新后，分发器同时通知桌面形态和输入法形态。两个出口各自维护一个渲染适配层（Renderer Adapter），将同一个 EmotionState 映射为不同的视觉表达。

**交付物**：EmotionCore 独立模块 + 事件总线扩展接口 + 双出口分发器骨架。

### Phase 1 · 输入法迷你形态 MVP（3 周）

**目标**：角色以 48×48 到 64×64 头像出现在输入法界面，能显示基础表情变化。

#### 1-1 迷你形态视觉设计

迷你形态只渲染角色面部的极简版本。在 48px 空间内，只有三个视觉通道能被用户辨识：

**眼睛状态（信息密度最高）**。映射 EmotionState.mood.eye 参数：

| eye 参数值 | 迷你形态表现 | 触发场景 |
|-----------|-------------|---------|
| normal    | 两个小圆点，自然状态 | idle 默认 |
| half      | 两条短横线，半闭眼 | 深夜陪伴 / 困了 |
| wide      | 稍大的圆点 | 用户打了感叹号 / 惊讶 |
| sparkle   | 圆点+微小高光 | 好感度事件触发 |
| gaze偏移  | 圆点偏左或偏右 | 偷看用户方向 |

**腮红程度**。映射 EmotionState.mood.blushCheek，简化为 3 档：

| blushCheek 值 | 迷你形态表现 | 视觉实现 |
|--------------|-------------|---------|
| 0 ~ 0.2     | 无腮红 | 默认 |
| 0.2 ~ 0.6   | 微腮红 | 两颊添加 2px 粉色圆点 |
| 0.6 ~ 1.0   | 明显腮红 | 两颊粉色区域扩大+加深 |

**一句话气泡**。不是每次打字都弹出，只在特定事件触发时出现。气泡从迷你头像旁浮出，停留 3-5 秒后淡出。台词直接复用角色设计文档中的好感度分级台词系统。

气泡触发条件表：

| 事件类型 | 触发条件 | 示例台词（随好感度等级变化） |
|---------|---------|--------------------------|
| 长时间打字 | 连续输入 > 60 分钟 | L1:「该休息了。」 L3:「眼睛会累的。」 L5:「我帮你倒杯水？」|
| 深夜打字 | 23:00 后仍在输入 | L1:「……」 L3:「你又熬夜。」 L5:「别太晚了，嗯？」|
| 打字停顿后恢复 | 停顿 > 5min 后恢复 | L2:「回来了。」 L4:「我还在。」|
| 快速输入 | 打字速度 > 阈值 | L2:「（默默注视）」 L4:「很忙的样子。」|
| 日历事件 | 午餐/下班时间 | 复用现有事件通知台词 |

#### 1-2 迷你形态渲染实现

技术方案取决于目标平台：

**方案 A · Electron/桌面端自研输入法插件**。如果 DeskBuddy 本身就是桌面端 Electron 应用，迷你形态可以作为一个悬浮小窗口实现，定位锚定在系统输入法候选栏附近。用 Canvas 或 SVG 渲染 48px 头像。优势是完全可控，劣势是需要检测输入法候选栏的屏幕位置。

**方案 B · 系统托盘/任务栏常驻**。不直接嵌入输入法，而是在系统托盘区域放置角色迷你头像，用户打字时头像自动激活（有微动画），停止打字时渐入休眠。这是最轻量的实现，不依赖任何输入法 API。

**方案 C · 搜狗/百度输入法皮肤 SDK（远期）**。如果未来与输入法厂商合作，可以通过皮肤 SDK 将角色嵌入候选栏。但这需要商务合作，不作为 MVP 考虑。

**MVP 推荐方案 B**，原因：实现成本最低，不依赖第三方 API，可以快速验证「迷你形态跟随打字行为」这个核心假设。

#### 1-3 输入行为感知层

MVP 阶段只做**打字节奏监听**，不读取输入内容（隐私安全）。

实现方式：注册系统级键盘事件监听器，只采集以下信号：

- 键盘按下事件的时间戳序列（不记录具体按键）
- 计算：每分钟按键频率、连续输入时长、停顿间隔
- 生成 TypingEvent 发送到事件总线

```typescript
class TypingBehaviorSensor {
  private keyTimestamps: number[] = [];
  private sessionStart: number | null = null;
  private idleTimer: NodeJS.Timeout | null = null;
  
  onKeyDown() {
    const now = Date.now();
    this.keyTimestamps.push(now);
    
    // 开始新会话
    if (!this.sessionStart) {
      this.sessionStart = now;
      this.emit({ type: 'typing_start' });
    }
    
    // 重置空闲计时
    if (this.idleTimer) clearTimeout(this.idleTimer);
    this.idleTimer = setTimeout(() => {
      const duration = now - this.sessionStart!;
      this.emit({ type: 'typing_stop', idleDurationMs: 5 * 60 * 1000 });
      this.sessionStart = null;
      this.keyTimestamps = [];
    }, 5 * 60 * 1000); // 5分钟无输入视为停止
    
    // 检测长时间连续输入
    if (this.sessionStart && now - this.sessionStart > 60 * 60 * 1000) {
      this.emit({ 
        type: 'long_session', 
        minutesContinuous: Math.floor((now - this.sessionStart) / 60000) 
      });
    }
    
    // 检测深夜打字
    const hour = new Date().getHours();
    if (hour >= 23 || hour < 5) {
      this.emit({ type: 'late_night_typing', hour });
    }
  }
}
```

CPU 开销要求：idle 状态 < 0.1% CPU，活跃打字时 < 0.5% CPU。键盘事件回调只做时间戳记录，分析逻辑在定时器中批量执行（每 30 秒一次）。

**交付物**：可运行的迷你形态小窗口 + 打字节奏感知 + 基础表情响应。

### Phase 2 · 双形态联动（2 周）

**目标**：桌面形态和迷你形态的状态完全同步，互动可以跨形态累积。

#### 2-1 状态同步规则

两种形态从 EmotionCore 读取同一份状态，但各自有不同的渲染策略：

| 情感事件 | 桌面形态表现 | 迷你形态表现 |
|---------|------------|------------|
| blushCheek 升高 | Spine 脸红动画 + 耳尖变红 | 3档腮红色块 |
| combo 触发 | 完整反应链（freeze→blush→shy_turn→台词） | 眼睛变 sparkle + 短气泡 |
| 好感度升级 | 全屏特效 + 剧情动画 | 眼睛闪烁 + 特殊气泡「……！」|
| 事件通知 | 完整台词气泡 + 动画反应链 | 一句话浓缩气泡 |
| 情感衰减 | blush/gaze 连续衰减动画 | 渐变回默认表情 |

#### 2-2 注意力分配协议

关键设计：两个形态不应该同时争抢用户注意力。

**桌面在前台时（可见）**：迷你形态降为「回声模式」——只做最轻微的同步（跟着眨眼），不弹气泡，不做醒目动画。用户的注意力应该在桌面角色上。

**桌面被遮挡/最小化时**：迷你形态升为「代言人模式」——承担角色全部的存在表达责任。气泡触发频率适当提高，眼睛动画更活跃，成为角色在用户视野中的唯一窗口。

检测方式：桌面形态窗口监听 `visibilitychange` 事件或 `blur`/`focus` 事件，通过 EmotionCore 广播当前可见性状态给迷你形态。

#### 2-3 跨形态互动累积

迷你形态上的轻触互动应该写入 EmotionCore，和桌面形态的 hitbox 互动产生同等效果：

| 迷你形态互动 | 等效桌面互动 | EmotionCore 效果 |
|------------|------------|-----------------|
| 点击迷你头像 | 点击面部 hitbox | +2 好感度，触发 combo 计数 |
| 长按迷你头像 | 摸头 hitbox | +5 好感度（L3+ 才响应） |
| 双击迷你头像 | 连续点击面部 | combo +1，触发对应反应 |

这样用户在输入法里「顺手戳一下」角色，回到桌面时角色状态已经同步更新。打通两个形态的互动积累，避免用户觉得「两个角色」。

**交付物**：双形态状态同步 + 注意力分配协议 + 跨形态互动累积。

### Phase 3 · 存在感系统（2 周）

**目标**：迷你形态有自己独特的情感节奏——不是桌面形态的缩小版，而是角色「在你余光里」时的独特表达。

#### 3-1 存在感衰减曲线

这是迷你形态独有的情感维度，桌面形态不需要。

设计原理：角色刚出现在迷你窗口时表情活跃（有微动画、眼睛会追踪），如果用户连续打字但始终没有和角色互动（没有点击、没有 hover），角色的「存在感」应该逐渐降低——

```
存在感等级（Presence Level）:

  Active ──→ Idle ──→ Dim ──→ Ghost
  (1.0)     (0.7)   (0.4)   (0.15)
  
  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐
  │活跃表情│→│安静待机│→│半透明 │→│几乎隐形│
  │眼睛追踪│  │偶尔眨眼│  │无微动画│  │只剩轮廓│
  │微动画  │  │无追踪  │  │      │  │      │
  └──────┘  └──────┘  └──────┘  └──────┘
   0-2min    2-5min    5-15min   15min+
```

**「回神」瞬间**：当角色已经衰减到 Dim 或 Ghost 状态时，用户突然点了一下——角色从半透明状态「回神」，眼睛微微睁大，有一个 0.3 秒的「被你看到了」的微表情。这个瞬间是迷你形态最有情感价值的互动时刻，完美贴合陆知行「被你注意到时有点不好意思」的人设。

```typescript
interface PresenceState {
  level: 'active' | 'idle' | 'dim' | 'ghost';
  opacity: number;        // 0.15 ~ 1.0
  lastInteraction: number; // timestamp
  surprisedOnWake: boolean; // 回神动画标记
}

function updatePresence(state: PresenceState): PresenceState {
  const elapsed = Date.now() - state.lastInteraction;
  
  if (elapsed < 2 * 60 * 1000) return { ...state, level: 'active', opacity: 1.0 };
  if (elapsed < 5 * 60 * 1000) return { ...state, level: 'idle', opacity: 0.7 };
  if (elapsed < 15 * 60 * 1000) return { ...state, level: 'dim', opacity: 0.4 };
  return { ...state, level: 'ghost', opacity: 0.15 };
}

function onMiniAvatarClicked(state: PresenceState): PresenceState {
  const wasDimOrGhost = state.level === 'dim' || state.level === 'ghost';
  return {
    level: 'active',
    opacity: 1.0,
    lastInteraction: Date.now(),
    surprisedOnWake: wasDimOrGhost, // 触发「回神」动画
  };
}
```

#### 3-2 打字节奏共鸣

迷你形态可以对用户的打字节奏产生微妙的视觉共鸣，而不需要读取内容：

| 打字模式 | 迷你形态反应 | 情感解读 |
|---------|------------|---------|
| 匀速稳定输入 | 角色眼睛偶尔看向用户方向 | 「在认真工作呢」|
| 突然快速连打 | 角色微微抬眉（eye: wide 0.3秒） | 「怎么突然紧张起来了」|
| 打字→长停顿→打字 | 角色做了一个小小的「回头看」 | 「在思考什么？」|
| 连续删除（退格频率高） | 角色轻微歪头 | 「在纠结怎么写？」|
| 打字突然停止 | 等待3秒后角色做「？」微表情 | 「怎么不打了」|

实现要求：所有反应延迟 0.5-1.5 秒（不能和按键完全同步，否则像机器监控），频率控制在每分钟最多 2 次微反应（不能太频繁打扰）。

**交付物**：存在感衰减系统 + 回神动画 + 打字节奏共鸣。

### Phase 4 · 内容层与成长体系（3 周）

**目标**：迷你形态不只是一个会动的头像，而是有内容深度的陪伴体验。

#### 4-1 情境台词系统

扩展角色设计文档中的台词系统，专门为输入法场景编写一套。台词按**好感度等级 × 情境类型**矩阵组织：

**工作陪伴类**（检测到持续打字时）

| 等级 | 台词池（随机选取） |
|-----|-----------------|
| L1  | 「……」「（看了你一眼）」|
| L2  | 「水喝了吗。」「该活动一下了。」|
| L3  | 「你专注的样子……算了，没什么。」「我在看书，不用管我。」|
| L4  | 「需要我帮你计时吗？」「……你工作认真的样子，我很喜欢。」|
| L5  | 「我帮你倒了杯水，别忘了喝。」「想你了。但你在忙，我不打扰。」|

**时间感知类**（与事件通知系统联动）

| 情境 | L1-2 | L3-4 | L5 |
|-----|------|------|-----|
| 早晨首次打字 | 「早。」| 「早安。今天天气不错。」| 「早安。昨晚睡得好吗？」|
| 午餐时间 | 「十二点了。」| 「该吃饭了，别忘。」| 「一起吃饭？我给你带了便当。」|
| 深夜 | 「……」| 「该睡了。」| 「再不睡我就关灯了。」|
| 周五下午 | — | 「快下班了。」| 「周末有什么计划？」|

**特殊成就类**（通过打字行为累积触发）

| 成就 | 触发条件 | 台词 |
|-----|---------|-----|
| 初次相遇 | 首次安装并开始打字 | 「你好。我是陆知行。」|
| 第100次互动 | 累计点击迷你头像100次 | 「我数过了，这是你第100次戳我。」|
| 连续7天陪伴 | 连续7天有打字活动 | 「一周了。习惯了有你在旁边。」|
| 深夜守护者 | 累计10次深夜陪伴 | 「又是深夜。你总这样……我也总陪着。」|
| 打字万字 | 单日打字量超过1万字符 | 「今天写了很多啊。辛苦了。」|

#### 4-2 迷你形态装扮系统

借鉴搜狗 AI 宠物的装扮自定义，但做得更克制、更贴合角色人设：

**季节/时间自动换装**（不需要用户操作）：

| 条件 | 装扮变化 |
|-----|---------|
| 冬季 | 迷你头像加围巾 |
| 夏季 | 眼镜换为墨镜 |
| 深夜 | 衣领变松/领带解开 |
| 雨天 | 头发微湿效果 |
| 周末 | 便服（去掉正式外套） |

**好感度解锁装扮**：

| 等级 | 解锁内容 |
|-----|---------|
| L2  | 微笑默认表情解锁 |
| L3  | 推眼镜小动画 |
| L4  | 「特别关注」模式——打字时角色看向你的频率增加 |
| L5  | 独占的深夜陪伴表情——困倦但坚持陪你的表情 |

#### 4-3 好感度增长与输入法场景的绑定

输入法形态为好感度系统新增以下增长途径：

| 行为 | 好感度增长 | 每日上限 |
|-----|----------|---------|
| 迷你头像点击 | +2 / 次 | 20 / 天 |
| 迷你头像长按（L3+） | +5 / 次 | 15 / 天 |
| 响应深夜关怀气泡后点击 | +10 | 1 / 天 |
| 连续陪伴天数 bonus | +3 × 连续天数 | — |
| 在「回神」瞬间点击 | +8（惊喜奖励） | 5 / 天 |

**交付物**：情境台词数据库 + 自动换装系统 + 好感度路径扩展。

### Phase 5 · 平台适配与性能优化（2 周）

#### 5-1 跨平台实现策略

| 平台 | 迷你形态实现方案 | 输入感知方案 |
|-----|---------------|------------|
| Windows | 系统托盘区域悬浮窗（WPF/Electron） | 全局键盘 Hook（低级别，不记录内容） |
| macOS | NSStatusItem 自定义视图 | CGEventTap（需辅助功能权限） |
| Linux | 系统托盘（libappindicator） | X11/Wayland 事件监听 |

各平台权限要求：

- **Windows**：无特殊权限，全局键盘 Hook 是标准 Win32 API
- **macOS**：需要「辅助功能」权限（System Preferences → Security & Privacy → Accessibility）。首次运行时需要引导用户授权，提供清晰的隐私说明
- **Linux**：X11 下无特殊权限；Wayland 下受限，可能需要使用 libinput

#### 5-2 性能预算

输入法形态必须比桌面形态更轻量，因为它是寄生在用户工作流中的：

| 指标 | 预算 | 实现策略 |
|-----|-----|---------|
| 内存 | < 15MB | 迷你头像用预渲染的 sprite sheet，不用实时 SVG |
| CPU（idle） | < 0.1% | 存在感 Ghost 状态暂停所有动画 tick |
| CPU（active） | < 0.5% | 表情变化用 CSS transition，不用 JS 逐帧 |
| 首次启动 | < 200ms | 头像资源预加载到内存 |
| 气泡弹出 | < 50ms | 台词预编译，不做实时 LLM 调用 |

#### 5-3 资源管理（借鉴搜狗输入法皮肤引擎架构）

三级缓存策略：

- **L1 热缓存**：当前 Presence Level 对应的表情帧（Active 状态约 8 帧）——常驻内存
- **L2 温缓存**：所有表情变化帧（约 30 帧）——首次加载后常驻
- **L3 冷存储**：装扮素材、特殊动画——按需从磁盘加载

目标：Active 状态下所有表情切换在 1 帧内完成（无加载延迟）。

**交付物**：三平台适配 + 性能达标 + 资源缓存系统。

### Phase 6 · 数据闭环与迭代（持续）

#### 6-1 关键指标定义

| 指标 | 定义 | 目标值 |
|-----|-----|-------|
| 迷你形态日活 | 每天至少看到迷你形态一次的用户占比 | > 80% |
| 气泡点击率 | 用户点击/关注气泡的比例 | > 15% |
| 跨形态互动率 | 在迷你形态互动后当天打开桌面形态的比例 | > 30% |
| 回神互动率 | Ghost→Active 的「回神」瞬间用户点击的比例 | > 20% |
| 好感度日增长 | 来自输入法形态的日均好感度增长 | 占总增长 30-40% |

#### 6-2 A/B 测试计划

| 测试项 | 变量 | 假设 |
|-------|-----|-----|
| 气泡频率 | 每小时 1次 vs 2次 vs 3次 | 2次/h 是最优平衡点 |
| 存在感衰减速度 | 快衰（5min→Ghost） vs 慢衰（20min→Ghost） | 慢衰减增加「被注意到」概率 |
| 回神动画强度 | 微妙（眼睛微张） vs 明显（+台词气泡） | 微妙版本更符合人设但数据可能更低 |
| 深夜关怀触发时间 | 23:00 vs 00:00 | 不同人群差异大，可能需个性化 |

## 四、风险与应对

**隐私合规风险**。打字节奏监听可能被用户视为「键盘记录器」。应对：首次启动时明确告知只采集按键频率和间隔、不记录任何具体按键内容或输入文字，提供一键关闭选项，在设置面板中可视化展示「我们采集了什么 / 没有采集什么」。

**性能投诉风险**。任何寄生在工作流中的常驻进程都可能被用户归咎为系统卡顿的原因。应对：严格执行性能预算，Ghost 状态下几乎零 CPU；提供「专注模式」一键暂停所有迷你形态活动。

**情感疲劳风险**。气泡台词重复率过高会让用户从「有陪伴感」变成「烦」。应对：台词池容量 > 200 条，使用加权随机（最近使用过的权重降低），同一台词 72 小时内不重复。

**双形态割裂风险**。如果同步不及时，用户在迷你形态戳了角色，但桌面形态没反应，会觉得是两个角色。应对：EmotionCore 状态变更走 IPC 广播，延迟 < 50ms；迷你形态互动后，桌面形态下次可见时做一个微妙的「刚被戳过」余韵表情。
